# 產品概念：記帳 App 的存活形式與演進

---
## 核心論述

隨著 AI Agent 發展，產品核心價值從高效率的輸入工具轉型為高維度的財務規則引擎。App 將從資料編輯器 Data Editor 演化為邏輯除錯器 Logic Debugger。

這不僅是功能的增加，而是本質上將軟體工程中的宣告式編程 Declarative Programming 概念應用於個人財務管理。

---
## *典範轉移:*

- **舊時代：注意力經濟 Attention Economy**
    - **核心行為:**
        - 手動開啟 App
        - 高頻點擊分類與輸入
        - 追求極致的操作手速
    - **產品定位:**
        - **資料編輯器 Data Editor**
        - **命令式思維 Imperative:** 針對每一筆錯誤的帳務結果進行手動修正，這是治標
        - **使用者角色:** 會計 Data Entry Clerk
- **新時代：意圖經濟 Intent Economy**
    - **核心行為:**
        - 使用者發出意圖 Intent
        - 系統自動感知並執行
        - 追求零點擊與自動化
    - **產品定位:**
        - **邏輯除錯器 Logic Debugger**
        - **宣告式思維 Declarative:** 針對產生錯誤帳務的規則進行修正，這是治本
        - **使用者角色:** 規則架構師 Architect 或 老闆 Boss

---
## *架構變革：Antigravity 邏輯模型*

此模型靈感源自 Python 的 Antigravity 概念：不直接對抗引力修改結果，而是改變引力規則修改代碼。

- **運作模式:**
    - **Code 即記帳紀錄:**
        - 記帳紀錄不再只是靜態資料，而是由一連串動態規則 Rule 運算後的產物
        - 當 OS 傳來帳務更新時，App 根據當前的財務規則自動生成紀錄
    - **Code Review 式的審視:**
        - 使用者定期開啟 App，不再是為了記帳，而是進行 Code Review
        - 檢查 AI 生成的流水帳是否符合預期
    - **非破壞性修正 Logic Debugging:**
        - 當發現帳務分類錯誤例如：轉帳給媽媽被記成轉帳支出
        - 使用者不直接修改該筆帳務的金額或分類
        - 使用者與 AI Agent 對話，修改背後的邏輯：這個帳號是媽媽的，以後都是孝親費
        - Agent 更新規則庫，並自動重構 Refactor 相關的歷史帳務
- **核心互動機制:**
    - **對話即編程 Prompt Engineering:**
        - 使用者與 AI 的對話過程，本質上就是在編寫或修正 App 的核心程式碼規則
        - 將自然語言轉譯為可執行的邏輯判斷式
    - **時光機效應 Replay & Simulation:**
        - 當規則改變修改 Code 後，系統具備回溯能力
        - 模擬新規則對過去 30 天或所有歷史數據的影響，並詢問使用者是否套用

---
## *雙層操作介面:*

- **例外處理 Override:**
    - 針對單次、偶發的偏差進行快速修正
    - 系統標記為例外，不影響全域規則
- **規則定義 Rule Definition:**
    - 針對重複性模式建立持久化規則
    - 提供可追溯性 Traceability 檢視類似 Git Blame
    - 點擊帳務可顯示：這筆帳被記為『娛樂』，是因為偵測到關鍵字『Steam』規則來源：12/05 的對話

---
## *開發重心轉移:*

- **規則引擎 Rule Engine:**
    - 建立高彈性 IF-THEN 邏輯庫
    - 支援複合條件：時間、地點、金額、關鍵字、帳號
- **記憶管理 Context Store:**
    - 采用 RAG 檢索增強生成架構
    - 儲存使用者偏好、關係網絡與歷史決策模式
- **決策儀表板 Dashboard:**
    - 介面功能從輸入轉為審視
    - 提供資產預測與模擬功能
