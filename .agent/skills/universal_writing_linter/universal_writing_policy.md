# Universal Writing Policy 通用寫作政策

## 核心核心精神與適用範圍
- **絕對無差別約束:** 本寫作政策為最高指導原則，不論檔案類型或用途，完全適用於**所有具備人類可讀註解之純文本檔案**，包含但不限於 `.md` 規格文件、各式原始碼如 `.ts`、`.tsx`、`.py`、`.cs`、`.go` 等中的 JSDoc 或單行多行註解、設定檔如 `.yaml` 等，**更嚴格適用於所有呈現於 UI 上的純文字內容**，包含元件內的字串、標籤、按鈕文字等。
- **優雅的平鋪直敘:** 寫作的基本功在於將複雜的脈絡透過文字本身流暢表達。依賴任何標點符號的補充說明、小聰明的標記等，皆視為撰寫能力低落的體現。

## 強制自我審查機制
- **修改後自我覆核:** 在產出或修改任何包含但不限於 `SKILL.md` 核心技能檔、各類規格文件與程式碼寫作的最後落筆階段，**必須**主動將自己的產出物與本通用寫作政策進行嚴格的比對檢查。
- **違禁品掃描:** 必須刻意與主動地去掃描自己剛寫下的文字，是否殘留如括號、數字編號、預期外的有序列表、星號列表或額外備註等違禁字眼。
- **負責到底:** 凡是在自我審查中發現違反本政策，必須立刻重新組織語句，直到確認完全消滅違規項，才能結束任務或進入後續流程。

## 癌症級別禁令，沒有例外

### 絕對禁止任何括號與引號
- **定義:** 包含字面意義上的圓括號、方括號、各式引號、粗括弧等任何包覆性符號。
- **原因:** 括號常被濫用於補充說明，會破壞句子的流暢度與文法結構，導致閱讀體驗破碎。
- **強制解法:** 所有位於這些包覆性符號內的補充說明，都必須平鋪直敘，直接轉化為優雅的前後文完整敘述。
- **適用範圍:** 無論是內文、標題、程式碼中的解釋註解，或是 UI 渲染字串，皆適用此禁令。

### 絕對禁止編號與有序列表
- **定義:** 包含數字編號如 1、2 等、特定字眼的階段標示如 第一階段、第二步驟 等、英文字母編號如 A、B 等以及任何隱含順序性的條列項目。
- **原因:** 編號與有序列表是一個深埋的維護陷阱。若在後續維護中需要於中間插入新步驟或概念，會導致破壞性的連鎖修改成本。
- **強制解法:**
  - **視覺隱含:** 條列項目的由上至下、由左至右視覺順序，本身就已經隱含了物理或邏輯上的先後順序。
  - **具名步驟:** 若自然順序不足以完整描述操作階段或概念框架，應為該步驟或階層專屬命名而非編號。例如，與其寫 數字標號加法、數字標號乘法，不如直接宣告 加法、乘法 兩個具名步驟，並在段落頂端註明其業務邏輯的預期總體順序。
- **適用範圍:** 規格書步驟、教學文件的認知層級區分、程式碼中的多步驟執行註解、以及 UI 上的項目選單或導航按鈕文字皆適用。

### 絕對禁止星號列表
- **定義:** 嚴禁使用星號作為列表的項目符號。
- **強制解法:** 所有層級的無序列表，均必須統一使用減號作為列表起始符號。

### 絕對禁止備註關鍵字
- **定義:** 嚴禁在句首或句尾使用 備註、注意 等關鍵字進行額外的突兀補充說明。
- **強制解法:** 所有的邊界條件與額外說明都必須優雅地整合進主要的規格描述或程式邏輯解說文字中。

### 絕對禁止最終狀態詞
- **定義:** 嚴禁在標題或檔案名稱中使用 Final 或 最終版 等代表已再無法修改的狀態詞。
- **允許:** 使用明確的數字版本號如 V1、V2 是被允許的。

### 絕對禁止檔案直接引用
- **定義:** 嚴禁在內文中以外部連結或字面名稱直接引用其他檔案名稱，例如 依據 no2_product_definition.md。
- **原因:** 檔案名稱或編號變更時，會造成大量無效連結與維護成本。
- **強制解法:** 直接描述該主題，例如 依據產品定義 或 參見角色定義，讓讀者透過上下文與知識庫搜索自行連結理解，而非依賴定死的硬編碼檔名。

### 絕對禁止金錢符號
- **定義:** 嚴禁使用任何貨幣與金錢符號於範例解釋或 UI/註解 描述中。
