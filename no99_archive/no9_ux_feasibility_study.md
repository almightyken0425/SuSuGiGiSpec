# UX 可行性分析：行動裝置介面限制與解決方案

---
## 問題意識

使用者指出核心疑慮：**概念很美好，但手機畫面太小，如何實際執行？**
Tinder 式滑動、Diff 預覽、規則編輯等複雜功能，在 Desktop 寬螢幕上很容易設計，但在 Mobile Portrait 直式介面下面臨巨大挑戰。

本分析將針對三大核心功能提出 **Mobile-First** 的具體互動設計方案。

---
## 首頁架構 Home Screen Architecture

### 核心入口設計
在傳統的底部導航列 Bottom Navigation 右側整合 AI 助理入口，點擊後向上展開對話介面。

```
[手機直式首頁]                 [AI Chat 展開後 Full Screen]
+------------------+         +------------------+
|      Header      |         |      Header      |
|  總資產           |         | ---------------- |
|  $ 1,234,567     |         |                  |
|                  |         |                  |
| [ 近期消費圖表 ] |          |    [對話框]       |
| ................ |  Tap    |                  |
| ................ |  ---->  |                  |
|                  |         |                  |
| [ 最新交易紀錄 ]  |         |                  |
| 7-11      -$100  |         | ---------------- |
|                  |         |[Add    Banner][3]|
|                  |         |[Review Banner][7]|
+------------------+         | ---------------- |

|       [ AI ]     |         +------------------+
+------------------+
```

## 待確認流程 Pending Review

### 挑戰
- **單手操作負擔:** 如果累積了 50 筆待確認項目，連續滑動 50 次 Tinder 卡片會造成手指疲勞 Repetitive Strain。
- **資訊密度:** 卡片佔據全螢幕可能導致無法同時參照其他資訊，如帳戶餘額。

### Mobile 解決方案：Day-View Context

採用 **逐日紀錄清單 Daily Record List** 作為審閱介面的基底。
當使用者從 Chat 點擊 Banner 進入審閱模式時，顯示的不是孤立的卡片，而是將這些待確認項目 **嵌入** 到原本的逐日交易列表中。

#### UI 佈局 Wireflow

```
[逐日審閱清單 Daily View]
+------------------+
|      10/25       | Header: No Arrows
|                  |
| [Add Banner]     | Sticky
| [!] 3 New Items  |
|                  |
| Oct 25 Today     |
| [v] [New] Uber   | <-- NEW Highlight
|     $250         |
| [v] [New] Uber   |
|     $250         |
|                  |
| [ Existing Item ]| <-- Context
|                  |
| Lunch   $120     |
+------------------+
| [Approve] [All]  |
| [Reject ] [All]  |
|   <  Prev/Next > | Footer Nav
+------------------+
```

## 逐日審閱 Contextual Review
- **基底:** 與首頁一致的 `Group by Date` 列表。
- **新增 New:** 在當日列表中插入新項目，背景高亮 Green，左側預選 Checkbox。
- **舊有 Existing:** 顯示當日已存在的交易 Dimmed，提供上下文對照，例如確認是否重複記帳。

## 邏輯修正 Diff 預覽 Logic Diff

### 挑戰
- **孤立檢視:** 若只看變更的單筆項目，使用者可能忘記當天還有其他類似交易。
- **批量操作:** 規則變更通常影響多筆，需支援批量核准。

### Mobile 解決方案：Inline Diff

同樣採用 **逐日紀錄清單**，但在該筆交易原地展開 Diff，並提供完整批量操作。

#### UI 佈局

```
[Logic Diff 預覽頁面 Daily View]
+----------------------+
|      12/09           | Header: No Arrows
+----------------------+
| [Review Banner]      |
| Rule: Steam -> 訂閱  |
+----------------------+
| Dec 09               |
|                      |
| [v] [Modified]       | <-- Yellow Highlight
|     Steam Games      |
|     $ 450            |
|     ---------------- |
|     Category:        |
|     [~娛樂~] > [訂閱]| <-- Diff
|                      |
| [ Existing Item ]    |
| 7-11        $ 55     |
+----------------------+
| [Approve] [All]      |
| [Reject ] [All]      |
|   <  Prev/Next >     | Footer Nav
+----------------------+
```

- **呈現方式:**
    - **Inline Expansion:** 受影響的交易在列表中呈現展開狀態。
    - **Diff:** 在卡片內部顯示變更前後的欄位，例如分類從 A 變 B。
    - **上下文:** 上下仍保留該日的其他正常交易，確保時間軸完整性。
- **確認機制:**
    - 採用勾選批次確認，或左滑/右滑逐筆確認。

## 規則編輯 Rule Customization

### 挑戰
- **輸入困難:** 在手機虛擬鍵盤上編寫複雜邏輯 IF A AND B THEN C 極其痛苦。
- **語意模糊:** 純自然語言輸入可能導致 AI 誤判，使用者不知如何修正。

### Mobile 解決方案：填空式引導 Mad Libs UI

不讓使用者寫作文，而是讓使用者填考卷。

#### UI 佈局

使用 **Chips 膠囊按鈕** 與 **Input Fields** 混合的自然語言句子。

```
[新增規則介面]
+----------------------+
| 當...                |
| [ 交易對象 v ]       | <-- Chip 點擊選單
| contains             |
| [ Uber      ]        | <-- Input 點擊鍵盤
|                      |
| 自動設為...          |
| [ 交通/通勤 v ]      | <-- Chip 點擊選單
|                      |
| + 新增條件           |
+----------------------+
|                      |
|      [ 儲存 ]        |
+----------------------+
```

- **操作介面:**
    - 畫面顯示固定結構的句子，變數位置挖空。
    - **句子:** 當 **[交易對象]** 包含 **[輸入關鍵字]** 時，自動設為 **[選擇分類]**。
    - **互動:**
        - 點擊 **[交易對象]** -> 底部彈出選單 轉帳來源 / 商家名稱 /備註。
        - 點擊 **[輸入關鍵字]** -> 彈出鍵盤輸入 例如: "Uber"。
        - 點擊 **[選擇分類]** -> 彈出分類選擇器 例如: "交通"。
- **優勢:**
    - 限制輸入範圍，減少錯誤。
    - 不需要打大量文字。
    - 邏輯結構一目瞭然。

---
## 總結可行性評估

| 功能 | Desktop 邏輯 | Mobile 修正方案 | 可行性 |
| :--- | :--- | :--- | :--- |
| **待確認** | Tinder 全螢幕卡片 | **分群清單 + Bottom Sheet 卡片** | 高 |
| **Diff 預覽** | 左右 Split View | **上下堆疊 + 刪除線/高亮** | 高 |
| **規則編輯** | 自由文字/程式碼 | **Mad Libs 填空式 UI** | 中 需精心設計句子模板 |

**結論：** 通過上述 UI 模式的轉換，這些高階功能在手機上完全可操作，且符合單手操作的人體工學。
