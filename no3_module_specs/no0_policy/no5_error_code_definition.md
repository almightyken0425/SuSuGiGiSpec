# 錯誤訊息定義

## 核心原則

- **目的:** 建立一個統一、可擴展且易於維護的錯誤代碼系統
- **易讀性:** 錯誤代碼應具有一定程度的可讀性，便於開發人員快速定位問題
- **資訊:** 錯誤訊息應清晰，並在可能時提供使用者可行的解決方案

---

## 結構定義

- **格式:** `[模組]-[錯誤類型]-[序號]`
- **範例:**
    - `AUTH-VALIDATION-001`
    - `DB-CONNECTION-001`

---

## 模組 Module

- **用途:** 定義錯誤發生的主要功能區域
- **範例:**
    - `AUTH`: 使用者認證模組
    - `DB`: 本機資料庫模組
    - `SYNC`: 批次同步模組
    - `PAYMENT`: 付費與訂閱模組
    - `FORM`: 表單處理模組
    - `APP`: 應用程式級別錯誤

---

## 錯誤類型 Error Type

- **用途:** 定義錯誤的性質
- **範例:**
    - `VALIDATION`: 使用者輸入或資料驗證失敗
    - `NOT_FOUND`: 找不到請求的資源
    - `PERMISSION`: 權限不足
    - `CONNECTION`: 連線失敗
    - `TIMEOUT`: 操作逾時
    - `CONFLICT`: 資料狀態衝突
    - `UNKNOWN`: 未知的錯誤

---

## 序號 Serial Number

- **用途:** 唯一標識該模組與類型下的特定錯誤
- **格式:** 三位數數字
- **範例:**
    - `001`
    - `002`

---

## 錯誤訊息 i18n

- **行為:** 錯誤代碼不應直接顯示給使用者
- **查詢:**
    - 應使用錯誤代碼作為 i18n 的 `key` 來查詢對應的本地化錯誤訊息
- **範例:**
    - **代碼:** `AUTH-VALIDATION-001`
    - **i18n Key:** `errors.AUTH_VALIDATION_001`
    - **i18n 訊息:** 登入失敗，請檢查您的帳號或密碼
        - **備註:** 範例為 zh-TW

---

## 錯誤處理實踐

- **日誌 Logging:**
    - **行為:** 應記錄完整的錯誤代碼
    - **備註:** 便於開發人員追蹤
- **使用者提示 UI:**
    - **行為:** 顯示查詢到的 i18n 訊息
    - **禁止:** 絕不應在 UI 上顯示原始錯誤代碼
